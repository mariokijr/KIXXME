# KixxMe

A Spanish-language gay social/dating app: users build a profile, discover nearby people on a list and map, like and chat in real time, and block users they don't want to interact with.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/scripts run seed-stripe` — create/sync Stripe products & prices (requires Stripe connected)
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Architecture decisions

- **Two databases.** Core app tables (`profiles`, `likes`, `conversations`, `messages`), auth, storage, and realtime live in **Supabase** (PostgREST via `supabase-js`); this schema is NOT DDL-modifiable from this repo. Repo-owned tables (e.g. `blocks`) live in the separate **Replit Postgres** (`DATABASE_URL`) via `@workspace/db` Drizzle. See `.agents/memory/kixxme-dual-database.md`.
- **No cross-DB foreign keys.** Relationships between Replit-Postgres tables and Supabase tables are enforced in application code, not SQL joins (e.g. block filtering loads block sets, then filters Supabase rows in JS).
- **Block enforcement is centralized** in `artifacts/api-server/src/lib/blocks.ts` and applied at every surface that exposes another user (discover/map, favorites, conversation list/create, send/read messages, image upload, like).
- **Visibility (hidden ids) is centralized** in `artifacts/api-server/src/lib/visibility.ts`: `getVisibilityContext(viewerId)` returns `{ hidden, iBlocked }` where `hidden = iBlocked ∪ blockedMe ∪ deactivated`. Discover list, map, stats, and likes all filter by `hidden`; the future suspension model plugs in here as one more union member. See `.agents/memory/kixxme-map-discovery-scope.md`.
- **Calidad mínima de perfil.** `GET /profiles` (discover grid + map) only returns "complete" profiles: Supabase columns (avatar_url/age/city/bio) are NOT-NULL filtered DB-side **before** `.limit(200)`, then JS refines bio ≥ `MIN_BIO_LENGTH` (10) chars + role + looking_for — the latter two batch-loaded from Replit Postgres via `getProfileDetailsForUsers(ids)` (no N+1). The web onboarding (`profile.tsx handleSave`) gates completion on the same field set. `/profiles/stats` is intentionally **not** quality-filtered (it counts all registered users — real Supabase count minus self/hidden). Gallery cap = 4 (`photos.ts MAX_PHOTOS`); the photo DELETE route blocks removing the last remaining photo (`count<=1` → 400, exact msg "Debes mantener al menos una foto en tu perfil."). Photos are managed as **4 fixed slots** (slot 0 = main): POST upload, PUT `/profiles/me/photos/:id` (`replacePhoto` — upload-new-first → repoint row keeping `position`+`is_avatar` → sync `profiles.avatar_url` if main **before** best-effort old-object delete → rollback new object if the row update fails), PATCH `.../:id/avatar` (set main), DELETE. `profile_photos` is the single source of truth; the top profile avatar is **display-only** (no separate avatar upload / reorder) so `profiles.avatar_url` only ever changes via the main photo, never desyncing. **Minimum age 18** is enforced server-side in `PUT /profiles/me` (400), in the discover filter (DB `.gte("age",18)` + JS), and the onboarding form.
- **Map scope filtering.** Scopes (Cerca/Provincia=radius, España/Europa=bounding box, Mundo=none) are pre-filtered inside the Supabase query **before** `.limit(200)`, then refined in JS (haversine for radius); radius scopes return empty when the viewer lacks coordinates. Gold-priority sort is applied **only when a scope is present** (the map; secondary, stability-dependent); the unscoped grid Descubrir applies a verified→completitud priority instead (see **Descubrir priority ordering**).
- **Descubrir priority ordering.** `GET /profiles` layers priority as stable `Array.sort` passes applied weakest→strongest, so the **last** pass is the strongest key: base sort (recent/distance/online) < `completitud` < `is_verified` < plan/Gold (scope/map only). Net effect: the grid surfaces **verified profiles first**, then **more-complete** profiles, then the chosen base sort; the map keeps paid Gold/Plus priority on top with verified/completitud as secondaries. `completitud` = a 0–6 score from gallery size (`min(photos,4)` via the batched `getPhotoCountsForUsers` in `lib/photos.ts` — no N+1) + bio-length tiers (≥80→+2, ≥30→+1); every candidate already passes calidad mínima, so it rewards going *beyond* the minimum. Priority applies **within** the most-recent-200 candidate sample, not globally.
- **Contract-first.** Endpoints are defined in OpenAPI, then server validates with generated Zod and the client uses generated React Query hooks.
- **Billing/entitlement.** Subscriptions via Stripe Checkout. `profiles.plan` in Supabase is the entitlement source of truth, written **only** by the Stripe webhook. Stripe data + a `billing_customers` mapping live in Replit Postgres (`stripe-replit-sync` `stripe` schema). Price resolution is by `lookup_key`; tier switches cancel the superseded subscription. A **testing-only** `GOLD_TEST_EMAILS` env allowlist (comma-separated, case-insensitive, mirrors `ADMIN_EMAILS`) promotes listed accounts to Gold at the `getPlan` read layer **without** a subscription — read-only override, never writes `profiles.plan` or touches Stripe (resolved via `admin.getUserById`, cached 5 min). It enables every functional Gold gate (Live, support, visitors), but display-only surfaces that read `profiles.plan` directly (discover/map badges, premium page) still show the real tier. Remove the var + restart to revoke. See `.agents/memory/kixxme-stripe-billing.md`.
- **Admin/moderation boundary.** The real boundary is `requireAdmin` (case-insensitive `ADMIN_EMAILS` allowlist) on every `/admin/*` handler; the web `AdminRoute` wrapper + admin entry link (keyed off `isAdmin` from `GET /me/moderation`) are **UX-only**. `requireAuth` runs a moderation gate that 403s suspended/banned/removed users with `{code, until}`; pass `{ allowModerated: true }` only for own-status read, logout, and self-service account deactivation/deletion (a moderated user keeps the right to leave/erase). Moderation hides users everywhere via the single `getUnavailableIds()` (deactivated ∪ moderated) union in `visibility.ts` — including the direct conversation read paths (GET messages, POST read) which 404, not just list/send. See `.agents/memory/kixxme-moderation-admin.md`.
- **Reversible admin remove + cross-DB user directory.** Admin "remove" is a soft, restorable `removed` state on `account_moderation` (hidden everywhere via `getUnavailableIds()`, no expiry like a ban), **NOT a hard delete** — irreversible GDPR erase stays in self-service `deleteAccount`, which also purges the user's `moderation_actions` rows so history never outlives the account. Every sanction is logged append-only in `moderation_actions` from **inside** the `lib/moderation.ts` helpers (never the routes), so each transition is recorded exactly once. The admin user directory lists Supabase `profiles` but state lives in Replit Postgres (two DBs, no join): a status filter resolves the matching id set first (`getUserIdsInState` for suspended/banned/removed via `.in`; `getModeratedIds` complement for "active" via `.not in`) and constrains the Supabase query so `count:exact` + offset paging stay correct — never JS-filter after `.range()`. Self-sanction is guarded (no sole-admin lockout) and the free-text search strips PostgREST `,()` grammar chars. See `.agents/memory/kixxme-moderation-admin.md`.
- **Admin notifications are derived, not stored.** Like user likes/matches, the admin "new report" notification has no table: `GET /notifications/summary` returns an `admin` block (`open_reports`/`open_flags`/`latest_report_at`) only for admins, mirroring `/admin/summary`'s open-row definitions; the frontend bell toasts when a new report arrives. See `.agents/memory/kixxme-moderation-admin.md`.
- **Verification badge is two-DB, approve writes Supabase first.** The badge itself is Supabase `profiles.is_verified` (the source of truth, like `plan`); `verification_requests` (Replit Postgres) is only the review queue. Approve writes `is_verified=true` in Supabase **before** flipping the Drizzle row to `approved`, so a mid-way failure leaves the request `pending` (admin retries) and `getVerificationStatus` derives "approved" whenever `is_verified` is true regardless of queue row — self-healing + idempotent re-approve. Review acts only on `pending` rows. A partial unique index (`userId` WHERE pending) prevents duplicate open requests.
- **Identity selfie is private, admin-only.** The verification request carries an identity selfie kept in a dedicated **private** Supabase bucket (`verification-selfies`, `public:false`), separate from public profile photos; the raw object never leaves the API — admins see it only via short-lived signed URLs minted inside `requireAdmin`-gated `GET /admin/verifications`. Mime allowlist is jpeg/png/webp (no SVG → no stored-XSS) with a 5MB decoded cap. Request ordering is cheap-checks → upload → insert; any insert failure best-effort removes the just-uploaded object, then **re-reads status** to tell the partial-unique race (→409) from a real failure (logged, →500) without sniffing the driver error. `deleteAccount` calls `purgeUserVerification` before the irreversible auth delete so a selfie never outlives its account.
- **Visitor identities are a paid feature (Plus + Gold), enforced server-side.** `GET /me/visitors` always returns the deduped `count` but only populates `visitors` when `plan !== "free"` (`can_see_visitors`); identities never leave the API for free users. Both count and list are filtered through `getVisibilityContext().hidden` (blocked ∪ deactivated ∪ moderated). This deliberately extends visitor identities to **Plus** (not Gold-only) — the premium page lists "Quién visitó tu perfil" under Plus.
- **KixxMe Live matchmaking.** Gold-only. The matcher runs in one transaction under a global `pg_advisory_xact_lock`, scans `FOR UPDATE SKIP LOCKED`, and atomically dequeues both users + inserts the call (no double-matching), storing both parties' filters + skip streak. Staleness/`RING_TTL` expiry is lazy on read; lifecycle transitions re-select `FOR UPDATE` with participant (IDOR) guards. Skip ("Siguiente") flips the call to `skipped` and re-queues **both** users (skipper streak+1, partner reset); 3 consecutive skips → 429. **Media plane = LiveKit** — `issueMediaToken()` mints a room-scoped, Gold-gated token (via `lib/livekit.ts mintRoomToken`) attached to the DTO only while `active`; `endCall` fires best-effort `deleteRoom` to close the 2h-token rejoin window after hangup/block. Frontend latches creds per `call.id` (`useLiveKitCall`) so the ~2s poll's token churn never reconnects. See `.agents/memory/kixxme-live.md`.

## Product

- Email/password auth (Supabase) with profile creation (bio, photos, location).
- Mandatory main photo + "calidad mínima": onboarding can't finish without a main photo plus age, city, a bio (≥10 chars), Rol/Preferencia and "Qué buscas" (toast lists what's missing). Photos use **4 fixed slots** on "Mi perfil" (slot 1 = "Principal" + 3 additional): empty slots show an "Añadir" placeholder, filled slots offer Cambiar / Elegir como principal / Eliminar. The gallery is capped at 4 and you can't delete your only photo (blocked with "Debes mantener al menos una foto en tu perfil."). Public profiles show a carousel of up to 4 photos with bio/edad/ciudad/rol/qué busca/estado/insignias. Incomplete profiles do not appear in Descubrir (grid + map) until those fields are filled. Minimum age is **18** (enforced server-side, in discovery, and in the form).
- Discover nearby users two ways from the "Descubrir" tab: a Tinder-style swipe deck ("Tarjetas", default — drag right=Me gusta, left=No me interesa, up/button=SuperLike, tap ℹ️ for photos+details) and the original grid ("Cuadrícula"); choice persists. Plus a world map with scope filters (Cerca/Provincia/España/Europa/Mundo) and live registered/online stats; Gold users get advanced filters (online/foto/verificados/edad) and priority visibility (crown markers on top), with a Gold upsell for everyone else. All surfaces are sorted/filtered by distance and online status. Descubrir also prioritizes **verified profiles first**, then **more-complete** profiles, ahead of the chosen sort.
- Like and SuperLike profiles, and view a favorites list. Free users get 10 likes/6h + 1 SuperLike/24h (Plus/Gold higher/unlimited); hitting a limit shows a full-screen "Hazte Premium" modal (`src/lib/limit-upsell.tsx`, not a toast) that blocks the action and links to `/premium`. A mutual like is a Match ("🎉 ¡Es un Match!") that opens chat. Free users are told when they receive a SuperLike but not by whom; Plus/Gold see the sender.
- One-to-one realtime chat with text, photo, and voice-note messages (photos open in a fullscreen zoom lightbox; voice notes record + play back with a duration readout), read receipts, and unread counts. Gold↔Soporte tickets support the same photo + voice-note attachments in both directions.

## Gotchas

- The API server has **no hot reload** — restart the `artifacts/api-server: API Server` workflow after backend edits. The web app uses Vite HMR.
- **Never call `signUp`/`signInWithPassword`/`refreshSession`/`setSession` on the shared service-role `supabase` client (or `supabaseAuth`)** — those attach the user's session to the singleton and demote every later `.from()` query to that user's RLS context process-wide (phantom "new row violates row-level security policy" errors + 0-row reads, only *after* a login). Use the dedicated `supabaseUserAuth` anon client. See `.agents/memory/kixxme-supabase-client-session-pollution.md`.
- Two separate databases (see Architecture decisions). You cannot `CREATE`/`ALTER TABLE` on Supabase from this repo; new repo-owned tables go in the Replit Postgres (`DATABASE_URL`) via Drizzle.
- After editing `lib/api-spec/openapi.yaml`, run `pnpm --filter @workspace/api-spec run codegen`. After editing a `lib/*` package, run `pnpm run typecheck:libs` before leaf typechecks.
- Chat image uploads are base64 in the request body; very large images can hit the Express body size limit (`PayloadTooLargeError`).
- Transactional emails send via the **Gmail connector** — until it is connected, `sendEmail` degrades gracefully (logs and returns false; signup/checkout/support still succeed). After connecting, wire the real client into `gmail.ts` (see `.agents/memory/kixxme-transactional-email.md`). Password-reset emails are handled by **Supabase Auth**, not this repo.
- Stripe Checkout will not load inside the Replit preview iframe — open it in a pre-opened tab from the click gesture, not a `window.location` redirect (see `premium.tsx`).
- Admin access requires the `ADMIN_EMAILS` env var (comma-separated allowlist). While it is empty, **no account is an admin** (no admin link, `/admin` redirects to `/discover`, all `/admin/*` endpoints 403). The API server has no hot reload — restart it after changing `ADMIN_EMAILS`. The client admin gate is UX-only; server `requireAdmin` is the real boundary.
- KixxMe Live uses **LiveKit** for video. The frontend drives the call state machine by polling `GET /live/state` (dynamic interval) — there is no socket/push; media creds (`mediaToken`/`mediaUrl`) ride on that poll and are latched client-side per `call.id` (see `useLiveKitCall`) so token churn never reconnects. Needs `LIVEKIT_*` secrets; the whole media path no-ops gracefully without them. Camera/mic only publish on the published domain (preview iframe lacks `allow=camera;microphone`).
